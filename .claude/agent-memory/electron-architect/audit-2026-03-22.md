### Mismatches Found

1. **`queue:status`** ‚Äî Registered in `handlers/llm.ts` but has NO corresponding preload bridge method. Dead handler; unreachable from renderer.

2. **`update:ready`** ‚Äî Main process sends this event on line 129 of `src/main/index.ts` when an update is downloaded, but:
   - No preload bridge exposes a listener for it
   - No renderer code subscribes to it
   - The auto-update notification is silently lost

3. **Duplicate preload files** ‚Äî There are TWO preload definitions:
   - `src/preload/entry.js` (plain JS, actually used at runtime)
   - `src/preload/index.ts` (TypeScript, provides types via `ClippiAPI` export)

   These must be kept manually in sync. They are currently in sync, but this is fragile. A single source of truth would be better.

## 2. Security Review

### Good Practices

- **`contextIsolation: true`** ‚Äî Confirmed in `src/main/index.ts:57`
- **`nodeIntegration: false`** ‚Äî Confirmed in `src/main/index.ts:58`
- **No `remote` module usage** ‚Äî Grep confirms zero usage across the codebase
- **CSP header present** ‚Äî Set in `src/renderer/index.html:6`:
  ```
  default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'
  ```
- **Proper IPC pattern** ‚Äî Uses `ipcRenderer.invoke` / `ipcMain.handle` (request-response), not the older `send`/`on` pattern for commands. The `on` pattern is only used for legitimate push events (watcher).
- **`safeHandle` wrapper** ‚Äî All IPC handlers go through a try/catch wrapper that serializes errors properly.

### Security Concerns

1. **`sandbox: false`** (line 59 of `src/main/index.ts`) ‚Äî Needed for preload `require()`, but weakens the security boundary. The comment acknowledges this. Consider migrating the preload to not need `require()` (use a bundled preload instead).

2. **Unsanitized file paths from renderer** ‚Äî MEDIUM RISK:
   - `import:folder` accepts `folderPath` from renderer and walks it recursively. No validation that the path is within expected directories. A compromised renderer could trigger reading any directory.
   - `import:analyze` accepts arbitrary `filePaths[]` from renderer.
   - `replay:openInDolphin` accepts `replayPath` from renderer and passes it to `spawn`. It does check `fs.existsSync(replayPath)` but no path validation/sandboxing.
   - `watcher:start` accepts `replayFolder` from renderer and starts a chokidar watcher on it.
   - `analyze:run` accepts `replayPaths[]` from renderer and reads those files.

   **Mitigation**: In practice, all these paths come from either the dialog picker (safe) or the config (user-set). But there is no server-side validation enforcing this.

3. **`config:save` accepts arbitrary keys** ‚Äî The handler accepts `Partial<Config>` but performs no validation. A compromised renderer could write unexpected keys to the config file. The `saveConfig` function just does `{ ...current, ...config }` spread.

4. **CSP `'unsafe-inline'` for styles** ‚Äî Minor. Allows inline style injection. This is common in Electron apps with React but could be tightened.

5. **CSP missing `connect-src`** ‚Äî The CSP does not specify `connect-src`, which defaults to `'self'`. This means the renderer cannot make network requests directly (good ‚Äî LLM calls are routed through main process). However, the `openrouter:models` fetch in main process is the correct pattern.

6. **`key.env` bundled in `extraResources`** ‚Äî The electron-builder config bundles `key.env` into the production app. This means API keys ship with the app binary. If this is intentional (shared team keys), it's a distribution risk. If developers build locally, they might accidentally include their personal keys.

## 3. Error Handling

### Global Safety Net

The `safeHandle` wrapper in `src/main/ipc.ts` catches all errors and re-throws as serialized `Error` objects. This prevents raw objects from failing IPC serialization and crashing the main process. **This is well-designed.**

### Handler-Level Analysis

| Handler | Internal try/catch | Notes |
|---|---|---|
| config.ts | Relies on `safeHandle` | `loadConfig` has its own try/catch for parse errors. OK. |
| dialog.ts | Relies on `safeHandle` | OK ‚Äî `dialog.showOpenDialog` shouldn't throw. |
| import.ts | Relies on `safeHandle` + internal walk try/catch | `walk()` catches per-directory errors. Good. |
| analysis.ts | Relies on `safeHandle` | `processGame` can throw on corrupt .slp files. Caught by wrapper. OK. |
| llm.ts | Relies on `safeHandle` | `fetch` can throw on network errors. Caught by wrapper. OK. |
| stats.ts | Relies on `safeHandle` | SQLite queries can throw. Caught by wrapper. OK. |
| watcher.ts | Relies on `safeHandle` | OK. |
| dolphin.ts | Relies on `safeHandle` | Has explicit `throw new Error` for missing paths. OK. |

### Potential Crash Vectors

1. **`spawn` in dolphin.ts** ‚Äî If `spawn` throws synchronously (e.g., ENOENT on the dolphin binary path despite `existsSync` check ‚Äî race condition), `safeHandle` catches it. If the spawned process crashes, `child.unref()` + `stdio: "ignore"` means it won't affect the main process. **OK.**

2. **Watcher `onError`** ‚Äî Errors in the watcher callback (`onImport`/`onError`) that send to the renderer via `webContents.send` could fail if the window is destroyed. `getMainWindow()?.webContents.send(...)` uses optional chaining. **OK.**

3. **`analyze:recent` SQL query** ‚Äî Uses parameterized query (`LIMIT ?`). Not vulnerable to SQL injection. **OK.**

## 4. Cross-Platform Concerns

### Path Handling

- **`path.join` used consistently** ‚Äî The codebase uses `path.join()` throughout `handlers/`, `config.ts`, `watcher.ts`, and `dolphin.ts`. No hardcoded `/` separators found in path construction.

### Dolphin Handler (`handlers/dolphin.ts`) ‚Äî Platform-Specific Issues

1. **Windows path construction** (line 33):
   ```typescript
   "C:\\Users\\" + require("os").userInfo().username + "\\AppData\\Roaming\\..."
   ```
   This is a hardcoded Windows path. Should use `path.join(require("os").homedir(), "AppData", "Roaming", ...)` instead. Currently works but is fragile.

2. **`which` command** (line 40):
   ```typescript
   execSync("which slippi-dolphin 2>/dev/null || which dolphin-emu 2>/dev/null", ...)
   ```
   Uses Unix-only `which` and shell redirection `2>/dev/null`. Guarded by `process.platform !== "win32"` check. **OK.**

3. **Slippi Launcher settings path** (line 90):
   ```typescript
   path.join(home, ".config/Slippi Launcher/Settings")
   ```
   This is Linux-specific (XDG config). On macOS, Slippi Launcher stores settings elsewhere (`~/Library/Application Support/`). On Windows, it's in `%APPDATA%`. **BUG: ISO detection will fail on macOS and Windows.**

4. **`spawn` with `detached: true`** ‚Äî Works on all platforms but has different semantics. On Windows, `detached: true` creates a new console window. This is probably fine for Dolphin.

### Watcher

- Uses `chokidar` with relative glob pattern `"**/*.slp"` and `cwd` option. Cross-platform safe. **OK.**

### Config/DB Paths

- `~/.magi-melee/` via `os.homedir()` ‚Äî Works cross-platform. **OK.**

## 5. State Management

### Global State (`src/main/state.ts`)

Simple module-level singletons:
- `mainWindow: BrowserWindow | null`
- `fileWatcher: { close: () => void } | null`

### Race Conditions

1. **Watcher start/stop race** ‚Äî The `watcher:start` handler calls `existing.close()` then `setFileWatcher(watchReplays(...))`. Since `watchReplays` is synchronous in its setup (chokidar watch creation is sync), this is safe. However, if two rapid `watcher:start` calls arrive, the first `close()` could race with the second's setup. In practice, IPC handlers run on the same event loop tick sequence, so this is unlikely. **Low risk.**

2. **Watcher `gameCount` increment** ‚Äî The `gameCount` variable in `watcher.ts` increments per file. Since chokidar events are sequential on the Node event loop, no race condition. **OK.**

3. **`closeDb()` called from watcher `close()`** ‚Äî The watcher's `close` function (line 120-122 of `watcher.ts`) calls `closeDb()`. This is also called from `window-all-closed` in `main/index.ts`. Double-close could be an issue if the watcher is still running when the window closes. The `window-all-closed` handler calls `watcher.close()` first (which calls `closeDb()`), then calls `closeDb()` again in `main/index.ts:148`. **Potential double-close bug.** Better-sqlite3's `close()` is idempotent, so it won't crash, but it's messy.

4. **Config read-modify-write** ‚Äî `saveConfig` does `loadConfig()` then writes merged result. If two concurrent `config:save` calls happen, last-write-wins with potential data loss. **Low risk** since config saves are user-triggered.

## 6. Build Configuration

### electron-builder (`package.json` "build" section)

**Issues found:**

1. **`node_modules/**/*` in files array** ‚Äî This bundles ALL node_modules into the app. electron-builder usually handles this automatically via dependency detection. Explicitly including everything means devDependencies might leak in, inflating the package size. However, electron-builder typically still prunes devDependencies. **Minor inefficiency concern.**

2. **`key.env` in `extraResources`** ‚Äî As noted in security section, API keys bundled into production builds. The filter `["key.env"]` means it only copies if the file exists, so builds without it won't fail.

3. **No `asar` configuration** ‚Äî Defaults to `asar: true` (good). Code is packed into an asar archive.

4. **Missing `afterSign` hook for macOS notarization** ‚Äî No notarization config. macOS builds will trigger Gatekeeper warnings.

5. **No `electronVersion` pinned** ‚Äî Relies on the installed version. Fine for local builds but could cause CI reproducibility issues.

6. **Main entry point**: `"main": "src/main/entry.js"` ‚Äî This is the correct entry for both dev and production via the conditional require pattern in `entry.js`.

### Platform Targets

- Windows: NSIS + Portable (x64 only) ‚Äî OK
- macOS: DMG (x64 + arm64) ‚Äî Good universal binary support
- Linux: AppImage + deb (x64 only) ‚Äî OK, but no arm64 Linux support

## 7. Native Modules

### better-sqlite3

- Listed as a production dependency. electron-builder should handle native module rebuilding via `@electron/rebuild` (present in package-lock.json).
- No explicit `externals` config in vite.config.ts ‚Äî this is fine because better-sqlite3 is only used in the main process (compiled by tsc, not Vite).
- The main process TypeScript is compiled by `tsc` and the output references `better-sqlite3` as a normal require. electron-builder packages the native addon from `node_modules`.

### @slippi/slippi-js

- Listed as a production dependency. Has native components (ubjson).
- Same pattern as better-sqlite3 ‚Äî only used in main process.
- The `@slippi/slippi-js/node` entry point is specified in CLAUDE.md as required.

### Potential Issue

- **No `rebuild` script** ‚Äî There's no explicit `electron-rebuild` or `@electron/rebuild` in scripts. electron-builder runs rebuild during packaging, but developers doing `npm install` may get Node.js-targeted native modules rather than Electron-targeted ones. This could cause `MODULE_NOT_FOUND` or ABI mismatch errors during development. A postinstall rebuild script would help.

### Low Issues
1. ~~**Dual preload files** (`entry.js` + `index.ts`) must be kept in sync manually~~ **RESOLVED** ‚Äî `entry.js` deleted, single TypeScript source compiled via `tsconfig.main.json`

2. `contextIsolation: true` + `nodeIntegration: false` ‚Äî correct security defaults
3. No `remote` module usage
4. CSP header present and blocks external scripts

8. Parameterized SQL queries (no injection risk)
